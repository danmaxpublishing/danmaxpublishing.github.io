Core Concepts

Photo Mode Pro is built around one component, one settings asset and a handful of small, replaceable pieces. This page explains how they fit together so the how-to guides make sense at a glance.

The controller¶

PhotoModeController (menu: Photo Mode Pro/Photo Mode Controller) is the single entry point. Add it to any GameObject — or drop in the generated prefab — assign a PhotoModeSettings asset, and you are done. The component is marked [DisallowMultipleComponent], and if a second controller appears in the scene it disables itself with a warning: there is always at most one active controller.

Scripts reach it through the null-safe static accessor:

using PhotoModePro;

PhotoModeController photoMode = PhotoModeController.Instance; // null when none is enabled

Instance is set in OnEnable and cleared in OnDisable, so it is safe across scene loads. See Events & API for the full scripting surface.

Enter / exit lifecycle¶

EnterPhotoMode() (or the configured toggle binding — F / View by default) does, in order:

  1. Records the current Time.timeScale, AudioListener.pause, cursor visibility, cursor lock state and the main camera's enabled flag.
  2. Pauses the game per PhotoModeSettings.pauseMode (TimeScaleZero by default, or Unscaled to keep the game running) and audio per audioPauseMode.
  3. Clones the camera tagged MainCamera into a free photo camera (carrying over URP/HDRP camera data) and disables the original.
  4. Creates the effects backend for the active render pipeline and applies the current adjustments.
  5. Hides your registered game UI canvases and spawns the photo mode UI prefab.

ExitPhotoMode() restores exactly what it found: the recorded timeScale, audio pause state, cursor state and main-camera enabled flag, destroys the photo camera and UI, and re-shows anything the visibility toggles hid. If the scene had no EventSystem when the UI opened, photo mode creates a matching one and removes it again on exit — the scene is left exactly as found. Cinemachine-driven cameras hand off and resume cleanly too; see Integrations.

Safety nets

Replacing or unloading a scene while photo mode is open auto-closes it cleanly (state restored first, an in-flight capture abandoned). Additive loads — streamed chunks — are harmless and do not close the session. If the controller itself is destroyed while open, an OnDestroy fallback still restores time scale, audio, cursor and everything photo mode spawned.

EnterPhotoMode() no-ops when already active or when no main camera exists; ExitPhotoMode() no-ops while inactive or while a capture is in flight.

The adjustments model¶

PhotoModeController.Adjustments is a plain PhotoModeAdjustments object holding every image tweak in natural units — no pipeline types anywhere:

Field Unit / range Neutral
exposureEv EV, −3 .. +3 0
contrast, saturation, temperature −100 .. +100 0
vignette, filmGrain 0 .. 1 0
depthOfFieldEnabled bool off
focusDistance meters 10
aperture f-number (applied clamped to f/0.7–f/16) f/5.6

Values are sticky for the play session: close and reopen photo mode and your look is still there. If you write fields from script, call controller.NotifyAdjustmentsEdited() afterwards — details in Image adjustments.

Effects backends and capabilities¶

Adjustments are pushed through IPhotoModeEffectsBackend (namespace PhotoModePro.Effects), a small render-pipeline abstraction. Four implementations ship in the box: URP volume, HDRP volume (physical-camera depth of field), Post Processing Stack v2, and a minimal Built-in fallback shader. Selection is automatic from the active render pipeline; force one via PhotoModeSettings.backendOverride if you must.

Each backend reports EffectCapabilities flags (Exposure, Contrast, Saturation, Temperature, Vignette, Grain, DepthOfField, TransparentCapture). The UI hides any control whose capability is missing, so players never touch a dead slider, and gamepad navigation never lands on one. Scripts read the same flags via PhotoModeController.ActiveCapabilities — even while photo mode is closed, where it returns a prediction for the backend that would be selected right now. Per-pipeline differences are tabulated in Pipelines.

The settings asset¶

PhotoModeSettings (Assets > Create > Photo Mode Pro > Settings) holds every tunable: key/gamepad bindings, pause and audio behavior, free-camera limits (speed, sphere clamp, collision, FOV range, roll speed), the effects backend override, capture defaults and the save path, and visibility options. Duplicate it per configuration and assign it on the controller — or swap it from script via the Settings property while photo mode is inactive.

Presets¶

A PhotoModePreset asset is just a named bundle of adjustment values. Six ship built in (Clean, Noir, Vivid, Warm, Cool, Vintage); the controller's Presets list decides what the Presets tab offers. ApplyPreset(preset) copies the values in; ApplyPreset(null) resets to the neutral "Clean" state. See Presets.

The UI prefab and theme¶

The photo mode UI is an ordinary UGUI prefab (PhotoModeUIController) that the controller instantiates on enter and destroys on exit. It has four categories — Camera, Effects, Presets, Capture — cycled with Q/E or LB/RB, full keyboard/gamepad navigation, and a single row of hotkey hint chips along the bottom edge of the screen. Every color, sprite, font, motion parameter and UI sound comes from one PhotoModeUITheme asset — swap it to reskin everything. See Theming.

Capture pipeline¶

CapturePhoto() (or P / the SNAP button) renders the photo camera into a RenderTexture at 1×, 2× or 4× the screen resolution (Supersample), with temporal anti-aliasing and dynamic resolution disabled for the shot so the output is artifact-free and exactly the requested size. The result is read back with AsyncGPUReadback where the platform supports it, encoded to PNG with a timestamped, collision-safe file name, and saved to {persistentDataPath}/Photos (or PhotoModeSettings.customSavePath). A thumbnail toast confirms the shot, and OnPhotoCaptured(path, thumbnail) fires for your own code. Heavy supersampled captures show a "PROCESSING…" badge that never appears in the photo. Full details in Capturing photos; timing numbers live in Performance.
