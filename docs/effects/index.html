Image Adjustments

The Effects tab exposes the full adjustment set. Every value lives in PhotoModeController.Adjustments (a PhotoModeAdjustments object) in natural, pipeline-agnostic units, and is sticky for the play session.

The sliders¶

Control Field Unit / range Notes
Exposure exposureEv EV, −3 .. +3 Relative exposure compensation; identical semantics on every pipeline (on HDRP it maps to post-exposure, so it never fights auto-exposure).
Contrast contrast −100 .. +100 0 is neutral.
Saturation saturation −100 .. +100 −100 is grayscale.
Temperature temperature −100 (cool) .. +100 (warm) 0 is neutral.
Vignette vignette 0 .. 1 Edge softness differs slightly per pipeline.
Grain filmGrain 0 .. 1 Response normalized across pipelines; subtle character differences remain.
Depth of Field depthOfFieldEnabled toggle The two rows below hide while this is off.
Focus Dist focusDistance meters, 0.1 .. 100 (slider) Logarithmic slider: half the travel covers the visibly focus-pulling 0.1–3 m. The readout stays in meters.
Aperture aperture f-number, f/0.7 .. f/16 (slider) See below.

The aperture slider¶

The aperture slider spans f/0.7 – f/16 with a perceptual log-stop mapping: each step of slider travel multiplies the f-number by a constant ratio, so a nudge near f/1 changes the bokeh about as much as a nudge near f/11 — the way photographers think in stops. Lower f-numbers mean a shallower focus plane and bigger bokeh; f/0.7 is the widest aperture HDRP's physical camera accepts, and beyond ~f/16 the blur is sub-pixel at game scales.

Clamping

PhotoModeAdjustments.aperture still accepts any positive value from script, and old presets keep loading unchanged — but the effect backends clamp into [PhotoModeAdjustments.ApertureMin, ApertureMax] (0.7–16) when applying. A preset saved at f/22 simply renders as f/16.

Per-pipeline availability¶

Not every backend supports every effect — the Built-in fallback (no Post Processing Stack v2) lacks temperature, grain and depth of field, and HDRP cannot do transparent captures. Unsupported controls hide automatically, and scripts can check the same flags:

using PhotoModePro;
using PhotoModePro.Effects;

bool hasDof = (PhotoModeController.Instance.ActiveCapabilities
               & EffectCapabilities.DepthOfField) != 0;

The full capability matrix and the visual differences between pipelines are on the Pipelines page.

Scripting adjustments¶

Write the fields directly, then tell the controller you did — that marks the effects dirty (re-applied next frame) and flips the preset label to "Custom":

using PhotoModePro;

var controller = PhotoModeController.Instance;

controller.Adjustments.exposureEv = 0.5f;
controller.Adjustments.saturation = -100f;   // grayscale
controller.Adjustments.vignette = 0.35f;
controller.Adjustments.depthOfFieldEnabled = true;
controller.Adjustments.focusDistance = 2.5f; // meters
controller.Adjustments.aperture = 1.4f;      // f-stops

controller.NotifyAdjustmentsEdited();

Warning

Without NotifyAdjustmentsEdited() your changes sit in the model but never reach the screen until something else (a slider, a preset) triggers a re-apply. The UI sliders call it for you; direct field writes must.

To replace the whole set at once, use a preset instead — ApplyPreset(...) copies values, re-applies and raises OnAdjustmentsReplaced so the UI re-seeds its sliders. See Presets.

Values for capabilities the active backend lacks are simply ignored — never clamped or destroyed — so a look authored on URP survives a session on the Built-in fallback intact.
