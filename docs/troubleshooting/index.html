Troubleshooting

Symptom-driven fixes, including every historical bug fixed in 1.2.x–1.4.0. First steps for almost anything: update to the latest version, run Tools > Photo Mode Pro > Setup Wizard (one-click fixes for pipeline, input, TMP and demo-material problems), and run Tools > Photo Mode Pro > Run Self-Test.

Version check

Many entries below were bugs fixed in a specific release. The current version is listed in the Changelog; if you are behind, updating is usually the whole fix.

The demo renders pink/magenta¶

Symptom: The demo scene (or its props) shows pink/magenta materials.

Cause: The demo materials were generated for a different render pipeline than the one currently active — pink is Unity's "shader not supported" color.

Fix: Since 1.4.0 the demo is pipeline-aware: materials are generated for the active pipeline, and opening the demo scene under a mismatched pipeline offers "Rebuild demo for current pipeline" (your choice can be persisted with don't-ask-again). Accept it, or run Tools > Photo Mode Pro > Build Demo Assets manually. On 1.3.0 and earlier the demo was URP-only; update the package. The Setup Wizard's demo-materials check flags this too.

TextMeshPro text is missing or shows boxes¶

Symptom: UI labels are invisible or render as empty squares.

Cause: The TMP Essential Resources are not imported in your project. The photo mode UI uses TextMeshPro.

Fix: Import them once via Window > TextMeshPro > Import TMP Essential Resources, or use the Setup Wizard's one-click fix (it checks TMP essentials). The demo builder imports them automatically.

No EventSystem in my scene¶

Symptom: You worry the UI needs an EventSystem your scene doesn't have (or, on old versions, the panel didn't respond to input without one).

Cause: UGUI input requires an EventSystem; not every game scene has one.

Fix: Fixed in 1.3.0; update the package. Photo mode now creates an EventSystem on open if the scene has none (matching your active input handling), logs a single info message, and removes it on exit. The scene is left exactly as it was found. If you prefer, add your own EventSystem and photo mode will use it.

Photo mode opens but the game keeps running¶

Symptom: Entering photo mode doesn't freeze gameplay.

Cause: The pause behavior on your PhotoModeSettings asset is set to the non-pausing "unscaled" mode. Or your gameplay code runs on unscaled time, so timeScale = 0 doesn't affect it.

Fix: Check the pause behavior on the settings asset assigned to your PhotoModeController (the default pauses via timeScale = 0). If your game loop uses unscaled time deliberately, gate it yourself from the controller's OnEntered/OnExited events; see Events and API.

Audio doesn't resume (or doesn't pause) after photo mode¶

Symptom: Game audio stays paused after exiting photo mode, or never pauses on enter, typically in games that drive audio through mixer snapshots.

Cause: PhotoModeSettings.audioPauseMode controls this. The default, AudioListenerPause, sets AudioListener.pause on enter and restores the prior value exactly on exit; if your own code also writes AudioListener.pause, or your audio runs through snapshot transitions, the two systems fight.

Fix: Set audioPauseMode to None (the package then never touches audio) and transition your own snapshots from the events:

controller.OnEntered += () => pausedSnapshot.TransitionTo(0.1f);
controller.OnExited  += () => gameplaySnapshot.TransitionTo(0.1f);

Upgrading from 1.2.x

audioPauseMode replaced the old pauseAudio bool in 1.3.0. Projects that had unchecked pauseAudio must select None once on their settings asset; see the Upgrade Guide.

Gamepad navigation goes dead after a mouse click¶

Symptom: Arrow keys / D-pad do nothing when photo mode opens, or stop working after clicking a control with the mouse, until you switch tabs with Q/E. (NB-01)

Cause: Mouse clicks clear the EventSystem selection by design (so the focus ring never lingers for mouse users); older versions had no recovery path, so navigation lost its anchor.

Fix: Fixed in 1.3.0. Update the package. The first control of the active tab is auto-selected on open, and pressing a navigation input while nothing is selected re-selects it, so navigation never goes dead. If it persists: call PhotoModeUIController.EnsureSelection() from your own code, and report it via Tools > Photo Mode Pro > Report a Bug.

Gamepad right stick or triggers don't work¶

Symptom: Right-stick look and trigger fly up/down do nothing on gamepad.

Cause: You are on the legacy Input Manager. The default InputManager.asset defines no axes for the right stick or triggers, and the package never edits ProjectSettings. This is a documented limitation rather than a bug.

Fix: Install and enable the Input System package (recommended), or add the missing axes to your own InputManager.asset. Keyboard + mouse is fully supported on legacy.

Focus rings are cut off at panel edges¶

Symptom: The thin focus ring around toggles, preset buttons, Fly/Orbit, 1x/2x/4x or the capture-folder row is clipped by the side panel's scroll mask. (NB-03–NB-07, NB-09, NB-10; regressed and re-fixed as NB-16, NB-17)

Cause: Focus rings extend a few pixels beyond each control; without enough inner content padding the scroll viewport's mask clips them. The 1.4.0 regression had a subtler cause: the hover/selection scale-up (theme hoverScale) pushed rings on full-width controls past the Phase 3 padding.

Fix: Fixed in 1.3.0 and finally in 1.4.0 (padding is now computed from the ring size and the theme's selection scale, and regeneration-proof regression tests guard it). Update the package. If it persists: you are likely using a custom theme with a large hoverScale; reduce it, or rebuild the UI prefab via Tools > Photo Mode Pro > Build Demo Assets.

Keyboard/gamepad selection scrolls out of view¶

Symptom: Arrowing down a long tab (e.g. Effects) moves the selection below the visible scroll area and the panel doesn't follow. (NB-02)

Cause: Older versions had no scroll-follow logic for EventSystem selection changes.

Fix: Fixed in 1.3.0; update the package. The panel now auto-scrolls minimally, just enough to reveal the selected control plus its focus ring. It never moves when the selection is already visible.

The highlight overlaps neighboring controls or hides toggle checkmarks¶

Symptom: A soft blue glow bleeds past controls, overlaps the bottom menu / presets / 1x-2x-4x row, lingers after mouse clicks, hides toggle checkmarks, or sits on the wrong supersample button. (Phase 2 items 2–5, 13, 27)

Cause: The original expanding soft-glow highlight extended 8–12 px past controls and persisted after clicks.

Fix: Fixed in 1.2.1. Update the package. The glow was replaced by a thin ~2 px snug outline ring (circular on knobs/toggles, rounded-rect on buttons/tabs) that appears only during keyboard/gamepad navigation; mouse clicks clear the selection.

The Effects panel overflows / mouse wheel zooms instead of scrolling¶

Symptom: Long tabs run off the panel with no way to scroll, and the mouse wheel zooms the camera even when the pointer is over the UI. (Phase 2 items 1, 9)

Cause: Pre-1.2.1 the side-panel pages had no scroll view and wheel input went to camera zoom unconditionally.

Fix: Fixed in 1.2.1. Update the package. All pages live in a shared ScrollRect with a slim scrollbar; the wheel scrolls the panel when the pointer is over any UI and zooms the camera otherwise. Wheel zoom also updates the FOV slider live (two-way sync).

The aperture slider seems to do nothing above f/3¶

Symptom: Dragging the aperture slider up changes bokeh only at the low end; most of the travel produces no visible difference. (NB-15)

Cause: The old slider range extended far past visually useful f-stops, mapped linearly, so almost all the travel sat in a perceptual dead zone.

Fix: Fixed in 1.4.0. Update the package. The slider is re-ranged to f/0.7–f/16 with a perceptual log-stop mapping, so the full travel produces visible depth-of-field change. Out-of-range values from old presets or scripts clamp when applied (no API break; the property still accepts any positive value). See the Upgrade Guide.

The Focus Dist slider seems to do nothing above ~3 m¶

Symptom: Focus distance visibly pulls only at the low end of the slider. (Phase 2 item 15)

Cause: The slider mapped 0.1–100 m linearly; the visibly focus-pulling 0.1–3 m range occupied a sliver of the travel.

Fix: Fixed in 1.2.1; update the package. The slider now uses a logarithmic mapping (half the travel covers 0.1–3 m); the readout stays in real meters.

Focus Dist and Aperture rows show while Depth of Field is off¶

Symptom: The DoF sub-controls remain visible with the DoF toggle off. (Phase 2 item 16)

Cause: A UI bug. The rendered blur deactivated correctly, but the rows did not hide.

Fix: Fixed in 1.2.1. Update the package.

Slider changes don't show in HDRP¶

Symptom: On HDRP, the Focus Dist and Aperture sliders have no effect. (Phase 2 items 37, 38)

Cause: FOV smoothing wrote Camera.fieldOfView every frame, which silently resets usePhysicalProperties, so HDRP's physical-camera depth of field never saw the values.

Fix: Fixed in 1.2.1. Update the package. The camera is now driven via focalLength while physical properties are in use. If a custom camera script of yours writes fieldOfView on the photo camera every frame, it will reintroduce the problem; drive focalLength instead on physical cameras.

Film grain or vignette looks different across pipelines¶

Symptom: The same slider value reads much stronger or weaker on one pipeline (originally: grain far stronger on PPv2 than URP, vignette far weaker on PPv2 than URP). (Phase 2 items 30, 31)

Cause: The underlying post stacks respond differently to identical parameter values.

Fix: Fixed in 1.2.1. Update the package. URP grain uses a coarse lookup with low luminance response to match the PPv2 look, and PPv2 vignette response is boosted toward URP's. Subtle character differences remain by nature of the stacks; see Pipelines.

Effect controls are missing on the Built-in pipeline¶

Symptom: Temperature, film grain and depth of field don't appear in the UI on Built-in.

Cause: com.unity.postprocessing (Post Processing Stack v2) is not installed, so the minimal fallback backend is active. It supports exposure, contrast, saturation and vignette only, and unsupported controls hide automatically.

Fix: Install Post Processing Stack v2 for the full adjustment set. The Setup Wizard's effects-backend check points this out. See Pipelines for the capability matrix.

The Transparent BG toggle is missing on HDRP¶

Symptom: No transparent-capture option appears on HDRP.

Cause: By design. Clean alpha would require an alpha-capable color buffer format on the HDRP asset, which the package does not edit; the capability is reported as unsupported and the toggle hides.

Fix: None on HDRP. URP and both Built-in backends support transparent capture. See the FAQ and Pipelines.

Captured PNG has UI in it¶

Symptom: Your game's HUD appears in captured photos.

Cause: Photo mode hides only the canvases listed in the controller's Game Ui Canvases field; a HUD canvas not listed there (especially world-space or camera-space canvases the photo camera can see) renders into the shot.

Fix: Add your HUD canvases to Game Ui Canvases on the PhotoModeController; they are hidden on enter and restored exactly on exit. See Capture.

Capture flash and thumbnail don't appear with the UI hidden¶

Symptom: With Hide UI active, capturing works but you get no white flash, shutter feedback or thumbnail toast. (Phase 2 item 39)

Cause: Pre-1.2.1, Hide UI disabled the whole root canvas and took the capture feedback down with it.

Fix: Fixed in 1.2.1. Update the package. Capture feedback (flash, shutter sound, toast) is now exempt from Hide UI on every pipeline; only the chrome (toolbar, panel, hint bar) hides.

Capturing at 4x freezes the game with no feedback¶

Symptom: A 4x capture blocks for seconds and nothing indicates progress. (Phase 2 item 10)

Cause: Supersampled captures render synchronously on the main thread; older versions showed nothing during the block.

Fix: Fixed in 1.2.1. Update the package. 2x/4x captures now show a static "Processing…" badge for the duration (it never appears in the photo). The wall-time itself is expected — a 4x capture of a 1080p view renders at 7680×4320; see Performance.

Photo mode won't exit right after pressing capture¶

Symptom: Pressing the toggle immediately after capturing does nothing for a moment.

Cause: By design: ExitPhotoMode() no-ops while a capture's async GPU readback is in flight (a few frames), to avoid tearing down the photo camera mid-readback.

Fix: Wait the fraction of a second; the exit then proceeds normally.

Captures look slightly different from the on-screen image¶

Symptom: A capture is crisper than the live view, or you expected TAA in the shot.

Cause: By design. Temporal anti-aliasing (URP and HDRP) is disabled for the duration of every capture and dynamic resolution is forced off, because TAA history is invalid for an offscreen render and produces ghosting/jitter. MSAA/SMAA/FXAA are unaffected.

Fix: Nothing to fix. This guarantees artifact-free output at exactly the requested size.

Player pops or jumps after Hide Player¶

Symptom: Toggle Hide Player on and off, exit photo mode, and the character jumps or pops on return to gameplay. (NB-13)

Cause: Older versions hid the player with SetActive(false/true); re-activation reset CharacterController grounding and rebound the Animator. Additionally, a shared binding (e.g. Space as both fly-up and jump) could bleed into gameplay on the closing frame.

Fix: Fixed in 1.3.0. Update the package. Hide Player now disables only the player's Renderer components, so physics/animation state survives exactly (regression-tested for position/velocity/grounded equality). If your gameplay code shares bindings with photo mode, also adopt the documented exit-frame pattern:

var photoMode = PhotoModeController.Instance;
if (photoMode != null && (photoMode.IsActive || photoMode.ExitedThisFrame))
    return; // photo mode owns input this frame

The demo character shrinks or scales up during jumps¶

Symptom: The demo adventurer briefly becomes tiny then scales back up while jumping, or after Hide Player. (Phase 2 items 20, 22)

Cause: Imported animation clips keyed m_LocalScale, which snapped the model back to FBX units during the airborne state and on Animator rebind.

Fix: Fixed in 1.2.1; update the package. The animator now uses de-scaled copies of the clips. If it persists after updating, run Tools > Photo Mode Pro > Build Demo Assets to regenerate the demo.

Hint chips overlap my game's UI (or look stacked/cramped)¶

Symptom: The hotkey hint chips collide with other UI, or pile up as a tall stack in the bottom-left corner. Earlier: the chips were too small to read. (NB-11, NB-12, NB-14)

Cause: The chips were originally undersized (fixed in 1.3.0 with a ×1.5 scale-up into four short rows), and the 1.3.0 four-row block read as a cramped stack at some window sizes.

Fix: Fixed in 1.4.0; update the package. The chips now form a single row along the bottom screen edge (wrapping to a second row only on very narrow/portrait windows) and the toolbar is raised so the two can never intersect; verified at 16:9, 21:9 and 4:3. If your own HUD occupies the bottom edge during photo mode, add its canvas to Game Ui Canvases so it hides while photo mode is open.

Hint chips show the wrong keys after rebinding¶

Symptom: You changed hotkeys on the settings asset but the on-screen hints still show the old keys. (Phase 2 item 12)

Cause: Pre-1.2.1, hint text was not updated on hotkey changes.

Fix: Fixed in 1.2.1. Update the package. Changing a hotkey in the Inspector now offers (via an editor dialog) to update the matching hint chips in the UI prefab. If you declined the dialog or use a moved/duplicated prefab, change the key again to re-trigger it, or edit the chip labels on your prefab copy.

The NONE preset button is missing¶

Symptom: The Presets tab no longer has a NONE entry.

Cause: By design since 1.3.0: NONE was visually indistinguishable from Clean, so it was removed. Clean is now the first preset and the active default.

Fix: Nothing to fix. ApplyPreset(null) remains the programmatic reset and maps to the same state, with no API break. See Presets.

Clicking the capture toast does nothing¶

Symptom: The thumbnail toast after a capture isn't clickable. (Phase 2 item 6)

Cause: Pre-1.2.1, the toast blocked no raycasts.

Fix: Fixed in 1.2.1. Update the package. Clicking the toast opens the file browser with the new photo selected (Explorer reveal on Windows; folder open elsewhere). The save-path row on the CAPTURE tab is clickable too.

The FOV slider ignores my Min/Max FOV settings¶

Symptom: PhotoModeSettings.minFov/maxFov limit wheel zoom but the FOV slider still runs 10–120. (Phase 2 item 14)

Cause: Pre-1.2.1, the slider range was hardcoded.

Fix: Fixed in 1.2.1. Update the package. The slider range is read from the settings asset at UI bind time.

Sprint doesn't speed up camera roll¶

Symptom: Holding Shift speeds up movement but Z/C roll stays slow. (Phase 2 item 8)

Cause: Pre-1.2.1, the sprint multiplier applied to movement only.

Fix: Fixed in 1.2.1; update the package. Roll now uses the same sprintMultiplier. (1.2.1 also added the missing "Space/Ctrl · Up/Down" and "Z/C · Roll" hint chips, Phase 2 item 7.)

Hint text is hard to read over bright surfaces¶

Symptom: Floating hint labels, toolbar Q/E hints or the toast filename wash out over white/bright backgrounds; the demo HUD looked like plain text. (Phase 2 items 21, 25)

Cause: Scene-floating text had no outline, and the old demo HUD was a single legacy Text element.

Fix: Fixed in 1.2.1. Update the package. Scene-floating text now gets a thin black TMP outline + underlay, and the demo HUD is rebuilt from the photo-mode UI building blocks.

I can't find a camera collision toggle in the UI¶

Symptom: Camera collision can't be switched in-game.

Cause: By design. Collision is a settings-asset-only option (Enable Collision on PhotoModeSettings) with no in-game toggle.

Fix: Set it on the settings asset. See Camera.

My forced backend override is ignored¶

Symptom: You set PhotoModeSettings.backendOverride but a different backend runs, with a warning in the console.

Cause: The forced backend is unavailable in the project (e.g. forcing Hdrp in a URP project, or PostProcessingV2 without the package installed). An unavailable forced backend logs one warning and falls back to Auto.

Fix: Install/activate the matching pipeline or package, or leave the override on Auto; detection from the active render pipeline asset is reliable. See Pipelines.

The Profiler shows GC allocations while photo mode is open¶

Symptom: With the Profiler attached in the editor, sliders or idle photo mode appear to allocate a little every frame, despite the zero-GC claim.

Cause: Editor-only instrumentation. Probes show the bytes come from uGUI/TMP editor profiler markers and the measuring recorder itself: uGUI's Slider.Set alone allocates ~123 B/frame in editor play mode with no package code involved. The package's own pipeline (model write, backend apply, label SetText) allocates zero bytes above idle; allocation gates in the test suite assert this.

Fix: Profile a player build; the editor-only allocations vanish. Expect one warmup allocation on the first open (cache building), then zero per-frame GC. See Performance.

Still stuck?¶

  1. Run Tools > Photo Mode Pro > Run Self-Test. It writes PhotoModeSelfTest.json with your environment and check results.
  2. Use Tools > Photo Mode Pro > Report a Bug (or email support.dan.max@gmail.com) and attach PhotoModeSelfTest.json, plus your Unity version, render pipeline and reproduction steps.

The FAQ covers intentional behaviors that are sometimes mistaken for bugs, and the Support Matrix lists exactly what is verified where.
