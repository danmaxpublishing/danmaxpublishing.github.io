Events & Scripting

Everything in the UI is also available from script. The whole public surface lives on PhotoModeController (namespace PhotoModePro), reached via the null-safe static PhotoModeController.Instance.

Lifecycle events¶

C# event Signature Fires
OnEntered Action Photo mode entered.
OnExited Action Photo mode exited (state already restored).
OnPhotoCaptured Action<string, Texture2D> A photo was saved (path and thumbnail).
OnAdjustmentsReplaced Action The adjustment values were replaced wholesale (preset applied or reset).

Three of these also exist as serialized UnityEvents on the component for Inspector-only wiring: On Entered, On Exited and On Photo Captured (a PhotoCapturedUnityEvent, i.e. UnityEvent<string, Texture2D>). The C# event fires first, then the UnityEvent.

using PhotoModePro;
using UnityEngine;

var controller = PhotoModeController.Instance;

controller.OnEntered += () => Debug.Log("Photo mode open");
controller.OnExited += () => Debug.Log("Photo mode closed");
controller.OnPhotoCaptured += (path, thumb) => Debug.Log($"Saved {path}");

Enter / exit from a pause menu¶

EnterPhotoMode(), ExitPhotoMode() and TogglePhotoMode() are plain public methods. Wire them to a button or call them from your game-state machine:

using PhotoModePro;

public void OnPhotoModeButtonClicked()
{
    pauseMenu.Hide();
    PhotoModeController.Instance.EnterPhotoMode();
}

void Awake()
{
    // Reopen the pause menu when the player leaves photo mode.
    PhotoModeController.Instance.OnExited += pauseMenu.Show;
}

Both methods are safe to spam: EnterPhotoMode() no-ops when already active (or when no main camera exists), ExitPhotoMode() no-ops while inactive or while a capture is in progress. IsActive and IsCapturing report the current state. See Integrations for the full pause-menu pattern.

Shared bindings: ExitedThisFrame¶

If a photo mode binding doubles as a gameplay action (the demo's Space is both fly-up and jump), the closing key press would otherwise bleed into gameplay on the exit frame. ExitedThisFrame is true only during that single frame:

void Update()
{
    var photoMode = PhotoModeController.Instance;
    if (photoMode != null && (photoMode.IsActive || photoMode.ExitedThisFrame))
        return; // photo mode owns input this frame

    // ... gameplay input ...
}

The demo's ThirdPersonCharacter uses exactly this pattern.

Audio: mixer snapshots instead of AudioListener.pause¶

By default photo mode sets AudioListener.pause on enter and restores the prior value exactly on exit. If your game drives audio through mixer snapshots, set PhotoModeSettings.audioPauseMode to AudioPauseMode.None (the package then never touches audio) and transition in your own handlers:

using PhotoModePro;
using UnityEngine.Audio;

[SerializeField] AudioMixerSnapshot pausedSnapshot;
[SerializeField] AudioMixerSnapshot gameplaySnapshot;

void Start()
{
    var controller = PhotoModeController.Instance;
    controller.OnEntered += () => pausedSnapshot.TransitionTo(0.1f);
    controller.OnExited  += () => gameplaySnapshot.TransitionTo(0.1f);
}

More audio strategies in Integrations.

Capability checks¶

ActiveCapabilities returns the EffectCapabilities flags of the active effects backend. While photo mode is closed, it returns a cheap, side-effect-free prediction of the backend that would be selected right now:

using PhotoModePro;
using PhotoModePro.Effects;

EffectCapabilities caps = PhotoModeController.Instance.ActiveCapabilities;

if ((caps & EffectCapabilities.DepthOfField) != 0)
{
    // safe to surface DoF in your own UI
}

bool canTransparent = (caps & EffectCapabilities.TransparentCapture) != 0;

Flags: Exposure, Contrast, Saturation, Temperature, Vignette, Grain, DepthOfField, TransparentCapture. The matrix per pipeline is on Pipelines.
