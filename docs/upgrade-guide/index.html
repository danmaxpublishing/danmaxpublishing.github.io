Upgrade Guide

How to move an existing project to a newer Photo Mode Pro version, newest first. Each step lists what changed, who is affected, and what (if anything) you must do. The full change list lives in the Changelog.

After any upgrade

  1. Let Unity reimport the package fully.
  2. Run Tools > Photo Mode Pro > Run Self-Test — it writes PhotoModeSelfTest.json and verifies your setup end to end.
  3. If you ship a modified copy of the demo, rebuild it via Tools > Photo Mode Pro > Build Demo Assets.

To 1.4.0¶

No breaking API changes. Two behavioral changes and new editor tooling:

Aperture range and mapping¶

The aperture constants changed: ApertureMin/ApertureMax are now 0.7/16 (previously 1/32), and the slider uses a perceptual log-stop mapping so its full travel produces visible depth-of-field change.

  • The aperture property still accepts any positive value; out-of-range values from old presets, saved settings or scripts clamp into f/0.7–f/16 when applied.
  • Presets you re-save automatically clamp to the new range.
  • Only act if your code or presets relied on apertures outside f/0.7–f/16: values below 0.7 now behave as f/0.7 and values above 16 as f/16.

UI layout: hint chips and toolbar¶

Hotkey hint chips now form a single row along the bottom screen edge (wrapping to a second row only on very narrow/portrait windows), and the toolbar is raised so the two bands cannot intersect.

  • No action if you use the generated UI prefab as shipped — it updates with the package.
  • If you modified the generated UI prefab, regenerate it (Tools > Photo Mode Pro > Build Demo Assets) and re-apply your customizations, or merge the layout changes manually.

New editor menus¶

Everything now lives under Tools > Photo Mode Pro: Setup Wizard, Add Photo Mode To Scene, Run Self-Test, Build Demo Assets, Open Documentation, Report a Bug, Welcome, and About. The Setup Wizard runs green/yellow/red checks (pipeline, effects backend, input handling, Cinemachine, TMP essentials, demo materials) with one-click fixes — run it once after upgrading.

The demo is also pipeline-aware now (no longer URP-only): opening the demo scene under a mismatched pipeline offers a one-click rebuild. A first-import Welcome window appears once per machine and is never re-shown across updates.

To 1.3.0¶

pauseAudio replaced by audioPauseMode¶

The pauseAudio bool on PhotoModeSettings was replaced by the audioPauseMode enum (AudioPauseMode.AudioListenerPause / AudioPauseMode.None).

  • Old serialized assets fall back to the default, AudioListenerPause — the same behavior as the old pauseAudio = true. No action for most projects.
  • Action required if you had unchecked pauseAudio: select AudioPauseMode.None once on your settings asset. With None the package never touches audio, so drive your own mixer-snapshot transitions from OnEntered/OnExited — see Events and API.

NONE preset button removed¶

Clean is now the first preset and the active default; the NONE button is gone because it was indistinguishable from Clean.

  • ApplyPreset(null) still works as the programmatic reset and maps to the same state — no API break.
  • See Presets.

Minimum Unity version drops to 2021.3¶

1.3.0 lowers the minimum from Unity 6000.0 to 2021.3 LTS (verified on 2021.3, 2022.3 and 6000.3, across URP/HDRP/Built-in and both input handlers). Purely additive — nothing to migrate; see the Support Matrix.

To 1.2.0¶

Breaking: IPhotoModeEffectsBackend (custom backend implementors only)¶

If you implemented your own effects backend, two interface changes apply:

public interface IPhotoModeEffectsBackend
{
    // New in 1.2.0: report what your backend can do.
    EffectCapabilities Capabilities { get; }

    // Changed in 1.2.0: Initialize() now receives the photo camera.
    void Initialize(Camera photoCamera);

    // ...unchanged members...
}
  • Add the Capabilities property (return the EffectCapabilities flags your backend supports — the UI hides controls for everything else).
  • Update Initialize() to Initialize(Camera photoCamera); backends that do not need the camera can simply discard the argument.

Projects that only use the built-in backends are unaffected — backend selection is automatic, with a manual PhotoModeSettings.backendOverride if you need to force one. See Effects and Pipelines.

Coming from 1.0.x or 1.1.x¶

Apply the steps above in order (1.2.0 → 1.3.0 → 1.4.0). In practice, unless you wrote a custom effects backend or unchecked pauseAudio, upgrading is: import, reimport, run the Self-Test, and rebuild the demo if you modified it. If anything misbehaves afterwards, start with Troubleshooting.
